# DentalCRM Backend — Product Plan

> Концептуальный план для создания тикетов в Jira.
> Без деталей реализации — только что делаем и зачем.

## Epic 1 — Users Module: Bootstrap (`/users/me`)

**Цель:** После логина фронт должен получать имя пользователя и название клиники, в которой он работает. Сейчас такого эндпоинта нет. Эндпоинт живёт в `UsersModule` — не в `AuthModule`, т.к. возвращает профильные данные, а не auth-токены.

---

### DENTAL-1 — Реализовать эндпоинт `GET /users/me`

**Тип:** Feature
**Приоритет:** High
**Оценка:** 2 SP

**Описание:**
Создать защищённый bootstrap-эндпоинт, который возвращает данные текущего авторизованного пользователя и его организации. Вызывается фронтом один раз при старте приложения — не на конкретной странице, а в корне SPA до рендера любого роута. Данные берутся из JWT-токена (userId, organizationId), без дополнительной передачи параметров.

**Ответ эндпоинта:**
```json
{
  "user": {
    "id": "uuid",
    "firstName": "string",
    "lastName": "string",
    "email": "string",
    "role": "owner | doctor | admin"
  },
  "organization": {
    "id": "uuid",
    "name": "string"
  }
}
```

**Acceptance Criteria:**
- [ ] `GET /users/me` доступен только авторизованным пользователям
- [ ] Возвращает данные пользователя из БД по `userId` из токена
- [ ] Возвращает название организации по `organizationId` из токена
- [ ] Один JOIN-запрос (не два отдельных SELECT)
- [ ] Пароль и служебные поля не попадают в ответ
- [ ] Swagger-документация добавлена

## Epic 2 — Patients: полный CRUD

**Цель:** Реализовать полный цикл работы с пациентами — создание, просмотр, редактирование, удаление, поиск и фильтрация.

**Текущее состояние:**
- ✅ `GET /patients` — список с поиском, фильтром по статусу, пагинацией, сортировкой
- ✅ `GET /patients/:id` — получение одного пациента
- ❌ `POST /patients` — отсутствует
- ❌ `PATCH /patients/:id` — отсутствует
- ❌ `DELETE /patients/:id` — отсутствует
- ❌ Фильтр по тегам — отсутствует
- ❌ Фильтр по полу — отсутствует
- ❌ Фильтр по дате рождения (диапазон) — отсутствует
- ❌ Управление тегами — отсутствует

### DENTAL-2 — Создание пациента `POST /patients`

**Тип:** Feature
**Приоритет:** High
**Оценка:** 3 SP

**Описание:**
Реализовать эндпоинт создания нового пациента. Пациент создаётся в рамках организации текущего пользователя (tenancy).

**Поля при создании:**
| Поле | Обязательное | Примечание |
|---|---|---|
| firstName | ✅ | |
| lastName | ✅ | |
| dateOfBirth | ✅ | формат date |
| gender | ❌ | male / female / other |
| phone | ❌ | |
| email | ❌ | валидация формата |
| addressStreet | ❌ | |
| addressCity | ❌ | |
| addressZip | ❌ | |
| addressCountry | ❌ | |
| tags | ❌ | массив строк |

**Acceptance Criteria:**
- [ ] `POST /patients` доступен только авторизованным
- [ ] `organizationId` берётся из JWT, не из тела запроса
- [ ] Валидация всех входных полей (class-validator)
- [ ] Email валидируется по формату, если передан
- [ ] Возвращает созданного пациента с `201 Created`
- [ ] Swagger-документация добавлена

---

### DENTAL-3 — Редактирование пациента `PATCH /patients/:id`

**Тип:** Feature
**Приоритет:** High
**Оценка:** 2 SP

**Описание:**
Реализовать частичное обновление данных пациента. Обновлять можно только пациентов из своей организации.

**Поля, доступные для редактирования:**
- firstName, lastName, dateOfBirth, gender
- phone, email
- addressStreet, addressCity, addressZip, addressCountry
- status (new / active / vip / archived)
- tags

**Acceptance Criteria:**
- [ ] `PATCH /patients/:id` доступен только авторизованным
- [ ] Нельзя редактировать пациента из чужой организации (возвращает 404)
- [ ] `organizationId` нельзя переопределить через body
- [ ] Передавать можно только нужные поля (partial update)
- [ ] Возвращает обновлённого пациента
- [ ] Swagger-документация добавлена

---

### DENTAL-4 — Удаление пациента `DELETE /patients/:id`

**Тип:** Feature
**Приоритет:** Medium
**Оценка:** 1 SP

**Описание:**
Реализовать удаление пациента. Удалять можно только пациентов из своей организации.

**Стратегия удаления:**
Soft delete (добавить поле `deletedAt`) — пациент не исчезает из БД физически, что важно для сохранения истории визитов и аудита.

**Acceptance Criteria:**
- [ ] `DELETE /patients/:id` доступен только авторизованным
- [ ] Нельзя удалить пациента из чужой организации (возвращает 404)
- [ ] Реализован soft delete (поле `deletedAt`)
- [ ] Мягко удалённые пациенты не появляются в `GET /patients`
- [ ] Возвращает `204 No Content`
- [ ] Swagger-документация добавлена

### DENTAL-5 — Расширить фильтры в `GET /patients`

**Тип:** Improvement
**Приоритет:** Medium
**Оценка:** 2 SP

**Описание:**
Дополнить существующий эндпоинт получения списка пациентов новыми фильтрами.

**Текущие фильтры (уже есть):**
- `search` — поиск по имени, фамилии, email, телефону
- `status` — фильтр по статусу
- `sortBy` / `sortOrder` — сортировка
- `page` / `limit` — пагинация

**Новые фильтры для добавления:**
| Параметр | Описание |
|---|---|
| `gender` | Фильтр по полу (male / female / other) |
| `tags` | Фильтр по одному или нескольким тегам (array contains) |
| `dateOfBirthFrom` | Пациенты, рождённые не раньше этой даты |
| `dateOfBirthTo` | Пациенты, рождённые не позже этой даты |
| `city` | Фильтр по городу |

**Acceptance Criteria:**
- [ ] Все новые параметры опциональны
- [ ] Фильтры комбинируются между собой (AND логика)
- [ ] `tags` поддерживает фильтрацию по нескольким тегам одновременно
- [ ] Валидация формата дат
- [ ] Swagger-документация обновлена

---

### DENTAL-6 — Управление тегами пациента

**Тип:** Feature
**Приоритет:** Low
**Оценка:** 2 SP

**Описание:**
Добавить специализированные эндпоинты для управления тегами пациента. Теги используются для категоризации (например: "VIP", "Аллергия на анестезию", "Постоянный клиент").

**Эндпоинты:**
- `POST /patients/:id/tags` — добавить теги к пациенту
- `DELETE /patients/:id/tags` — убрать теги у пациента

**Acceptance Criteria:**
- [ ] Нельзя добавить дублирующий тег
- [ ] При удалении несуществующего тега — не ошибка, а no-op
- [ ] Возвращает обновлённый список тегов
- [ ] Теги не чувствительны к регистру при поиске

---

### DENTAL-7 — Добавить фото пациента `POST /patients/:id/photo`

**Тип:** Feature
**Приоритет:** Low
**Оценка:** 3 SP

**Описание:**
Реализовать загрузку фото пациента. Фото хранится в облачном хранилище (S3 или аналог), в БД сохраняется только URL.

**Эндпоинты:**
- `POST /patients/:id/photo` — загрузить/заменить фото (multipart/form-data)
- `DELETE /patients/:id/photo` — удалить фото

**Acceptance Criteria:**
- [ ] Принимает только изображения (jpg, png, webp)
- [ ] Максимальный размер файла: 5 MB
- [ ] Старое фото удаляется из хранилища при замене
- [ ] `photoUrl` обновляется в записи пациента
- [ ] Возвращает новый `photoUrl`
